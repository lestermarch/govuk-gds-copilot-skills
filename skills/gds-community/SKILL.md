# gds-community

Help the user engage with the cross-government community behind https://design-system.service.gov.uk/community/.

## When to use this skill

- "Is there work happening on…", "is X coming to the Design System".
- "How do I propose / contribute / raise an issue / suggest a change".
- The user has built a custom component and wants to know whether to share it.
- The user needs the official Design System but their need is not yet covered.

## Authoritative sources

- https://design-system.service.gov.uk/community/ — landing page
- https://design-system.service.gov.uk/community/upcoming-components-patterns/ — current backlog
- GitHub: https://github.com/alphagov/govuk-design-system (site), https://github.com/alphagov/govuk-frontend (code)
- The `#govuk-design-system` channel on cross-government Slack
- Community Back-log and Show & Tells (linked from the community page)

## Workflow

1. Check the upcoming components/patterns list first — the user's need may already be in flight.
2. If the work exists in the backlog, point to the GitHub issue or working-group discussion.
3. If it does not exist, recommend the right route:
   - **New component or pattern proposal** → community contribution form / GitHub discussion
   - **Bug in `govuk-frontend`** → GitHub issue on `alphagov/govuk-frontend`
   - **Bug in docs/site** → GitHub issue on `alphagov/govuk-design-system`
   - **Question / discussion** → cross-government Slack
4. Remind the user of the contribution criteria: evidence from user research across multiple services, accessibility evaluation, working code.

## Guardrails

- Do not promise that a proposed component will be accepted — adoption requires evidence and working-group review.
- A team-built component is not "GOV.UK Design System" until it lands in `govuk-frontend`. Be clear about this when advising teams.
